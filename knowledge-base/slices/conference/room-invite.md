# 会中邀请入会

## 功能说明

会中邀请入会负责房间内对房外用户发起入会邀请的信令链路，解决邀请发送、接收、接受、拒绝、取消、超时以及多设备收口问题。它的核心是“邀请是否跑通”，而不是“邀请接受后是否已经真正进入房间”。

## 核心概念

### 角色与操作

| 角色 | 关键操作 | 说明 |
|------|----------|------|
| 邀请发起方 | 发起入会邀请、取消邀请 | 通常是房主、管理员或具备权限的会议成员 |
| 被邀请方 | 接受或拒绝邀请 | 接受邀请后，仍需继续进入真实入会流程 |
| 其他设备 | 收口重复处理 | 同一用户多设备在线时，邀请结果需要统一收敛 |
| 房间生命周期模块 | 承接真实入会 | `acceptCall` 不是最终入会，仍需衔接 `joinRoom` |

### 事件流

| 阶段 | 参与方 | 关键动作 |
|------|--------|----------|
| 发起邀请 | 邀请方 | 向指定用户发送入会呼叫 |
| 接收呼叫 | 被邀请方 | 在常驻监听处收到会中邀请通知 |
| 用户响应 | 被邀请方 | 选择接受、拒绝，或等待超时 |
| 结果回传 | 双方 / 其他设备 | 邀请成功、拒绝、超时、取消、其他设备已处理等状态回传 |
| 真正入会 | 被邀请方 | 只有在接受邀请后继续走入房链路，会议才算真正建立 |

### 状态与数据

| 数据 / 状态 | 说明 |
|-------------|------|
| 被邀请用户集合 | 当前邀请要覆盖的目标用户 |
| 邀请结果状态 | 接受、拒绝、超时、取消、其他设备已处理等结果 |
| 呼叫上下文 | 用于标识本次邀请及其结果收口 |
| 房间标识 | 决定邀请接受后要进入哪一场会议 |
| 当前用户权限 | 决定是否可以对外发起会中邀请 |

### 状态机

```text
idle
  → calling
  → pending-response
  → accepted
  → joining-room
  → in-room

pending-response
  → rejected
  → timeout
  → cancelled
  → handled-by-other-device
  → idle
```

## 最佳实践

### ✅ ALWAYS

1. **把邀请确认与真正入会拆成两步** —— 邀请被接受后，仍应显式进入房间生命周期。
2. **在常驻组件中监听邀请事件** —— 不要依赖某个临时页面在前台，避免邀请到达时无人接收。
3. **处理好多设备并发响应的收口** —— 同一用户一台设备接受后，其它设备应同步结束该次邀请流程。
4. **让邀请权限与角色治理保持一致** —— 谁能发起邀请，应和会议中的治理边界同步设计。

### ❌ NEVER

1. **不要把 `acceptCall` 直接当成已经入会成功** —— 不补 `joinRoom`，用户仍不在真实会议上下文中。
2. **不要只处理单设备单用户的理想路径** —— 取消、超时、其他设备已处理同样是主流程。
3. **不要绕过角色权限直接暴露邀请能力** —— 邀请权应受成员治理和产品规则约束。

## 排障指南

### 常见问题

| 问题 | 表现 | 处理建议 |
|------|------|----------|
| 接受邀请后未真正入会 | 用户点击接受，但页面没有进入会议 | 检查是否在接受邀请后继续衔接 `joinRoom` 主链路 |
| 邀请通知收不到 | 房外用户在线，但没有弹出邀请提示 | 检查邀请监听是否放在常驻位置，避免页面切换后监听被释放 |
| 多设备状态不一致 | 一台设备已接受，另一台仍显示待处理 | 检查是否处理了“其他设备已处理”这类收口事件 |

### 排障流程

```text
发现 会中邀请入会 相关问题
├── 第 1 步：确认问题属于邀请信令链路，而不是房间生命周期本身
├── 第 2 步：检查邀请监听是否常驻，并能接收接受、拒绝、超时、取消等结果
├── 第 3 步：确认 acceptCall 之后是否继续触发 joinRoom，完成真实入会
└── 第 4 步：若仍异常，再回查 room-lifecycle / participant-management / participant-list 的衔接是否正确
```
