# 会议配置与准入规则

## 功能说明

会议配置与准入规则描述的是“一场会议在进入真实房间流程前，带着什么基础信息和准入边界被定义”。
它主要负责 `roomId`、`roomName`、`password`、默认禁麦 / 禁画 / 禁共享 / 禁消息等初始化规则，以及房主在会中对房间名称、密码等基础信息的维护。

它关注的是**配置模型本身**，而不是“什么时候创建、什么时候加入、什么时候离开或结束会议”的流程判断：
- 创建 / 加入 / 离开 / 结束的主路径由 `room-lifecycle` 承接；
- 未来时间、参会人和提醒等排期能力由 `room-schedule` 承接；
- 会中临时禁言、关闭单成员设备等动态治理由 `room-moderation` 承接。

## 核心概念

### 角色与操作

| 角色 | 关键操作 | 说明 |
|------|----------|------|
| 会议发起人 | 定义房间基础配置 | 决定 `roomId`、`roomName`、密码和默认规则 |
| 业务服务端 | 分配稳定会议标识 | 在强管控场景中统一生成 `roomId`，并约束字段来源 |
| 房主（会中） | 维护房间基础信息 | 在已入会前提下更新房间名称、密码等基础配置 |
| 普通参会人 | 感知准入与默认规则 | 入会时遵守密码、默认禁麦等配置结果，并在 UI 上看到对应按钮禁用态与图标变化 |
| 预约会议模块 | 复用房间配置模型 | 在未来会议场景中复用房间基础配置，并额外补充时间和参会人 |

### 配置范围

| 配置域 | 典型字段 | 说明 |
|--------|----------|------|
| 房间标识 | `roomId`、`roomName` | `roomId` 是稳定主键，`roomName` 是展示名称 |
| 准入规则 | `password` | 控制会议进入门槛；真正的入会交互由 `room-lifecycle` 承接 |
| 默认入会规则 | 默认禁麦、禁画、禁共享、禁消息 | 决定成员进入房间后的初始受限状态 |
| 配置草稿 / 模板 | 表单状态、本地缓存、业务模板 | 用于在快速会议、预约会议等入口复用同一套字段语义 |
| 会中可维护基础信息 | 房间名称、密码等 | 由房主在会中通过房间信息更新接口维护 |

### 状态与数据

| 数据 / 状态 | 说明 |
|-------------|------|
| `roomId` | 房间唯一标识；更建议由业务后台统一生成 |
| `roomName` | 房间展示名称，影响会议列表、标题区等展示 |
| `password` | 密码准入字段；创建侧写入，加入侧在 `room-lifecycle` 完成输入与校验 |
| 默认会控规则 | 如默认禁麦、禁画、禁共享、禁消息等，决定成员初始入会状态 |
| 默认规则字段映射（Web） | `isAllMicrophoneDisabled`、`isAllCameraDisabled`、`isAllScreenShareDisabled`、`isAllMessageDisabled` | 对应全员静音、全员静画、全员禁止屏幕共享、全员禁止发送会中消息 |
| 配置草稿 | 创建前在表单或本地状态中维护的配置集合 |
| 配置校验结果 | 字段完整性、命名约束、密码策略等验证结果 |
| 可更新房间信息 | 会中允许房主维护的基础字段集合，如房间名称、密码 |

### 状态机

```text
draft
  → validating
  → ready-to-consume
  → consumed-by-lifecycle
  → maintained-in-room

ready-to-consume
  → consumed-by-schedule

maintained-in-room
  → info-updated

validating
  → invalid
  → draft
```

## 最佳实践

### ✅ ALWAYS

1. **优先让业务后台生成并下发稳定 `roomId`** —— 这样更容易保证会议标识唯一，也便于快速会议、预约会议和服务端创建会议共用同一套 ID 体系。
2. **把 `room-config` 当成创建输入模型，而不是流程判断器** —— 它回答“会议带什么配置被创建”，不回答“当前该走创建还是加入”。
3. **让快速会议、预约会议和服务端会议复用同一套字段语义** —— 避免 `roomName`、密码和默认规则在不同入口下产生不同含义。
4. **把密码房当成完整的准入链路设计** —— 创建侧要能设置密码，加入侧则由 `room-lifecycle` 提供输入、错误提示和重试闭环。
5. **把默认规则与会中动态会控分层** —— 默认禁麦 / 禁画 / 禁共享 / 禁消息属于创建期配置；在 Web 侧通常对应 `isAllMicrophoneDisabled`、`isAllCameraDisabled`、`isAllScreenShareDisabled`、`isAllMessageDisabled`。会中临时调整应交给 `room-moderation`。
6. **让默认规则同时驱动 UI 呈现** —— 例如开启 `isAllMicrophoneDisabled` 后，不应只限制开麦能力；麦克风按钮图标、可点击态和提示文案也应同步切到“全员静音 / 禁用态”。
7. **只在房主仍处于当前房间内时更新房间信息** —— 修改房间名称或密码属于会中的基础配置维护动作，不应脱离当前房间上下文执行。
8. **让预约会议复用房间配置，但不要把时间排期字段并入配置主语义** —— 时间、参会人和提醒依然应由 `room-schedule` 负责。

### ❌ NEVER

1. **不要在 `room-config` 里展开 `createAndJoinRoom` / `joinRoom` 的路径选择** —— 这是 `room-lifecycle` 的职责。
2. **不要把 `scheduleStartTime`、`scheduleEndTime`、参会人列表当成房间配置主语义的一部分** —— 这些是 `room-schedule` 的时间维度能力。
3. **不要把创建期配置和会中治理混成一个动作** —— 否则创建表单会沦为运行时会控的替代品。
4. **不要在房主已经离房后继续把房间信息更新当成可随时调用的全局配置接口** —— 它依赖真实的房间内上下文。
5. **不要让不同入口对同一个配置字段给出不同业务含义** —— 否则同一场会议会出现展示和行为不一致。

## 排障指南

### 常见问题

| 问题 | 表现 | 处理建议 |
|------|------|----------|
| `roomId` 策略混乱 | 不同入口下会议号来源不一致，容易冲突或难追踪 | 收口到统一的业务生成策略，并避免前端临时拼接高冲突 ID |
| 默认规则未生效 | 创建时已勾选默认禁麦 / 禁画，成员进房后却未体现 | 检查配置是否在创建阶段正确提交，而不是误写到会中治理逻辑 |
| 默认规则已生效但图标没变 | 用户被全员静音 / 禁画限制，但按钮仍显示可点击或图标仍像可用态 | 让房间默认规则同时驱动按钮禁用态、图标和提示文案，避免能力状态与界面状态分裂 |
| 密码房体验断裂 | 创建时设置了密码，但加入侧没有输入或错误提示 | 检查是否由 `room-lifecycle` 补齐密码输入、重试和失败提示闭环 |
| 快速会议与预约会议表现不一致 | 同样的配置在不同入口下表现不同 | 检查是否复用了同一套配置模型和字段语义 |
| 修改房间信息后 UI 不一致 | 房间名称或密码已更新，但部分界面仍显示旧值 | 检查房间信息更新后是否同步刷新房间级状态与相关展示区域 |

### 排障流程

```text
发现 会议配置与准入规则 相关问题
├── 第 1 步：确认问题属于房间基础配置，而不是创建 / 加入 / 离开主流程
├── 第 2 步：检查 `roomId`、`roomName`、`password` 与默认规则是否在提交前完成校验
├── 第 3 步：确认快速会议、预约会议和服务端会议是否复用了同一套配置字段语义
├── 第 4 步：若是密码房体验问题，回查 `room-lifecycle` 是否补齐加入侧交互
└── 第 5 步：若是时间、提醒或参会人问题，转查 `room-schedule`
```
