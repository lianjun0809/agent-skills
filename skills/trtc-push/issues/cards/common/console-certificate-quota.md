# 厂商证书数量 / 套餐上限

## 适用现象

当用户反馈以下现象时，优先使用本卡：

- 控制台新增 Push 应用或厂商证书失败。
- 报错包含 `cert limit exceeded`、`CertLimitExceeded`、证书数量已达上限。
- 用户询问体验版、标准版、高级版能添加多少证书。
- 多 App / 马甲包希望在同一 `SDKAppID` 下继续添加同厂商证书。

不适用于证书已存在但推送不达的运行时问题；那类问题应继续排查配置、注册链路和厂商回执。

## 共同根因

这是套餐 / 平台能力边界，不是 SDK Bug。不同套餐、平台和厂商对证书数量有上限；达到上限后
继续新增证书会被控制台拒绝。

## 必须收集的证据

- 当前 `SDKAppID`。
- 当前套餐版本（体验版 / 标准版 / 高级版等）。
- 控制台报错文案或错误码。
- 已配置的厂商、平台和证书数量。
- 用户是否存在多 App / 多包名 / 马甲包诉求。

## 排查步骤

1. 先判断用户是在“新增证书失败”，还是“已有证书推送失败”。
2. 如果是新增失败，确认报错是否为证书数量上限。
3. 核对套餐版本和当前已配置证书数量。
4. 判断用户诉求是增加同厂商证书、多平台证书，还是多个 App 共用同一应用。
5. 按套餐能力解释上限，不要把它当成配置错误或 SDK 接入问题。

## 解决方案

- 需要更多证书时，引导升级到支持更多证书的套餐。
- 多 App / 马甲包场景评估是否拆分 `SDKAppID` 或调整证书规划。
- 如果用户只是要替换证书，优先编辑/更新原证书，避免删除重建带来 `businessID` 变化。
- 对厂商侧自身限制，明确需要按厂商平台规则处理。

## 验证信号

- 用户能在控制台成功新增或更新证书。
- 新证书出现在正确 `SDKAppID` 和平台下。
- 客户端重新下载配置后，接入测试可查到对应证书 ID。
